Common installation issues

Almost every snag with Icarus Verilog on Windows comes down to a handful of small, fixable things. Here are the ones beginners hit most — what you'll see, why it happens, and how to fix it.

Windows

Common Icarus Verilog installation issues on Windows

Each issue below lists the symptom (what you'll see), the likely cause, and the fix. Work through them top to bottom — the first few solve the large majority of problems.

1. iverilog is not recognized as a command

Symptom. You type iverilog and the terminal replies:

'iverilog' is not recognized as an internal or external
command, operable program or batch file.

Cause. Windows can't find the program. When you type a command, Windows only looks in the folders listed in your PATH — a system setting that's just a list of folders to search. If Icarus Verilog's bin folder isn't on that list, the command "doesn't exist" as far as the terminal is concerned.

Fix. Easiest: re-run the installer and tick "Add executable folder(s) to the user PATH." To add it by hand, search Windows for "Edit the system environment variables" → Environment Variables → select Path → Edit → New, and add the bin folder (typically C:\iverilog\bin). Then open a new terminal — PATH changes only apply to terminals opened afterward.

2. vvp is not recognized as a command

Symptom. Compiling works, but running the simulation fails:

'vvp' is not recognized as an internal or external command...

Cause. The same PATH problem. vvp is the simulation runtime that runs the file iverilog produces, and it lives in the same bin folder.

Fix. Add that bin folder to your PATH (see issue 1) and reopen the terminal. Once it's on PATH, both iverilog and vvp work everywhere.

3. Forgetting to compile with SystemVerilog support

Symptom. Code that uses logic, always_comb, always_ff, or similar fails with a syntax error on a line that looks perfectly fine.

Cause. By default Icarus compiles to an older Verilog standard that doesn't include these SystemVerilog keywords.

Fix. Tell it to use the 2012 standard with -g2012: 

iverilog -g2012 -o sim tb.v design.v

4. Incorrect file extensions (.v.txt)

Symptom. Icarus can't find your file, or "your file" simply won't compile even though you can see it in the folder.

Cause. The file was saved as design.v.txt (common with Notepad). Windows hides known file extensions by default, so it still shows as design.v in Explorer.

Fix. In File Explorer, open the View menu and turn on File name extensions. Rename the file so it truly ends in .v. Better still, use a code editor like VS Code, which saves the real extension.

5. Missing source files during compilation

Symptom. An error naming a module you definitely wrote:

error: Unknown module type: full_adder

Cause. Your design is split across several files, but not all of them were listed in the compile command. Icarus only knows about the files you hand it.

Fix. Include every .v file the design uses: 

iverilog -g2012 -o sim tb.v alu_full.v alu_slice.v full_adder.v mux_2x1.v

6. Spaces in file paths

Symptom. The command fails or "can't open" a file when your project lives somewhere like C:\My Designs\alu.

Cause. The terminal treats a space as a separator, so it reads one path as several broken pieces.

Fix. Wrap any path containing spaces in double quotes — or avoid spaces in folder names altogether: 

iverilog -g2012 -o "C:\My Designs\sim" "C:\My Designs\tb.v"

7. Using Linux commands on Windows

Symptom. A tutorial says to run ./sim or make, and Windows answers '.' is not recognized or 'make' is not recognized.

Cause. Many guides assume Linux or macOS, where commands and paths look a little different.

Fix. The Windows workflow in Command Prompt or PowerShell is: compile, then run with vvp (no ./), then view the waveform: 

iverilog -g2012 -o sim tb.v design.v   &:: compile
vvp sim                                 &:: run the simulation
gtkwave dump.vcd                        &:: open the waveform

8. Antivirus or security software interference

Symptom. The compiled simulation won't run, vanishes right after compiling, or you see a message like "Operation did not complete successfully because the file contains a virus."

Cause. A freshly compiled, unsigned executable can trip overzealous security software, which quarantines or blocks it.

Fix. Add an exclusion for your project folder (and the Icarus install folder) in your antivirus settings, and restore the file from quarantine if it was removed. The simulation executables are safe.

9. Installing the wrong version

Symptom. Odd crashes, missing features, or is not a valid Win32 application.

Cause. An outdated build, or a version built for the wrong architecture (for example a 32-bit build on a 64-bit PC).

Fix. Install the latest 64-bit (x64) release from bleyer.org/icarus. If you previously installed an old version, uninstall it first.

10. Unsupported SystemVerilog features

Symptom. Errors on advanced verification constructs — class-based testbenches, constrained-random stimulus, some assertions, or UVM.

Cause. Icarus Verilog supports a large, very usable subset of SystemVerilog — more than enough for the designs here — but not every advanced verification feature found in commercial simulators.

Fix. Stick to the synthesizable subset (the style used across this site) and write straightforward testbenches. Nothing on Silicon From Scratch needs features Icarus lacks.

Before you ask for help

Quick troubleshooting checklist

Run through these first — they catch the vast majority of problems:

  • Did you open a new terminal after installing or editing PATH?
  • Do iverilog -V and vvp -V both print a version number?
  • Are you compiling with -g2012 for SystemVerilog code?
  • Did you list every .v file in the compile command?
  • Is the file really saved as .v (extensions visible), not .v.txt?
  • Are paths with spaces wrapped in "quotes"?
  • Are you running the sim with vvp sim (not ./sim)?
  • Is your antivirus leaving the compiled file alone?
  • Is this the latest x64 version from bleyer.org?
